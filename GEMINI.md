# Wedding Planner Hub — Project Instructions

This file serves as the primary source of truth for team-shared architecture, conventions, and workflows. Rigorously follow these instructions for every task.

---

## 🏛️ Architecture & Tech Stack

- **Framework**: Next.js 16+ (App Router)
- **Language**: TypeScript (Strict Mode)
- **Styling**: Tailwind CSS 4.x (using `@tailwindcss/postcss`)
- **Database**: Google Spreadsheets (Primary Data Store) via `googleapis`
- **Email**: Resend
- **SMS**: Twilio (US-based guests, TCPA compliance required)
- **UI Components**: Radix UI, Framer Motion, Lucide React
- **Maps**: Leaflet / React-Leaflet (for `VenueMap.tsx`)

---

## ⚠️ Critical Mandates

1. **Visual Integrity**: NEVER modify existing UI, design, styling, or frontend code unless explicitly asked. Preserve all existing components, colors, layouts, animations, and Tailwind classes.
2. **App Router Only**: All routes must live in `app/`. Never use the Pages router.
3. **Type Safety**: Always use TypeScript. Avoid `any` and prefer explicit interfaces/types for Google Sheet rows and API responses.
4. **Credential Security**: Never hardcode API keys or credentials. Use `.env.local` for local development. Never commit `.env` files.
5. **Idempotency**: API routes (especially RSVP and updates) should handle potential duplicate submissions or retries gracefully.

---

## 📋 Data Schema (Google Sheets)

### Sheet: "Guests"
Columns: `Id`, `Names`, `Phone`, `Email`, `Token`, `Sent`, `Clicked`, `Attending`, `PlusOnes`, `Table`, `Seats`, `OptOut`
- `Id`: UUID generated by the API.
- `Names`: Mandatory (Full name).
- `Phone`: Mandatory (10-digit US format preferred).
- `PlusOnes`: Mandatory (Numeric, defaults to 0).
- `Attending`: lowercase `yes` or `no`.
- `OptOut`: uppercase `TRUE` or `FALSE`.
- `Token`: Auto-generated 8-char unique string.
- `Sent`/`Clicked`: uppercase `TRUE` or `FALSE` (managed by marketing/admin).
- `Table`/`Seats`: Managed exclusively via the admin seating dashboard.

### Sheet: "Updates"
Columns: `Id`, `Date`, `Message`, `PhotoAlbumLink`, `SentTo`, `SentAt`

## 🛠️ Workflows & Conventions

### Global Loading State
- A custom `loading.tsx` is implemented at the root.
- It features a teal spinner, the couple's names, and "Opening Invitation..." message.
- It attempts to play `/public/sounds/loading-chime.mp3` during load transitions.

### API Implementation
- All API routes reside in `app/api/`.
- Use the utilities in `lib/` for Google Sheets, SMS, and Email.
- Prefer header-based column access for Google Sheets rather than hardcoded column letters.

### SMS Compliance (TCPA)
- Every SMS must include opt-out instructions (e.g., "Reply STOP to unsubscribe").
- Check the `OptOut` column in the "Guests" sheet before sending any message.

---

## 🧪 Testing Strategy

- **Verification**: Always verify API route changes by simulating requests (using `curl` or temporary scripts).
- **Data Integrity**: When modifying Google Sheets logic, verify that data is written to the correct columns without corrupting existing rows.
- **Visual Check**: Since frontend changes are restricted, any requested UI updates must be verified across mobile and desktop views.

---

## 📂 Directory Structure

- `app/admin/`: Protected admin routes for guest and seating management.
- `app/api/`: Backend logic and spreadsheet integrations.
- `lib/`: Shared utilities and service integrations (Sheets, SMS, Email).
- `components/ui/`: Reusable Radix-based UI components.
- `script/`: Maintenance and utility scripts (e.g., seeding).
